# Hexbin 全场对比热力图设计

## 1. 布局

用**一张完整的 SVG 画布**绘制整个篮球场俯视图（横向），左右各一个篮筐。选择器放在球场上方的独立行，左右各控制一个半场，公共控件置中。

```
                         页面布局（从上到下）

    [球员/球队 v] [赛季 v]   [固定标尺] [逐年播放]   [球员/球队 v] [赛季 v]
      左半场控制                中场控制                右半场控制

    单一 SVG 画布
    ◯ ────────── ── 跳球圈 ── ────────── ◯
    篮筐   左半场    中场     右半场    篮筐
          Hexbin              Hexbin
     hex 向右扩展              hex 向左扩展
          木地板底色 + 球场标线

    [投篮密度: 小 o --大小-- o 大    命中率: 浅色 方块 --深浅-- 方块 深色]
```

选择器栏位于球场上方，左右各一组（球员/球队下拉 + 赛季下拉），中间是公共开关和播放按钮。SVG 球场居中占据主区域，底部是颜色图例。

### 为什么用单一画布而不是两个独立面板

| 单一画布 | 两个独立面板 |
|---|---|
| 球场线条连贯（中线、边线一笔画完） | 中线被裁断，不自然 |
| 中场跳球圈、logo 等元素完整保留 | 无法展示中场元素 |
| 只有一个 SVG viewBox，缩放/导出方便 | 需要额外对齐 |
| 视觉上是"一张图"，不是"拼出来的" | 拼接感强 |

### SVG 层级结构

```
<svg viewBox="0 0 1880 500">
  <!-- 底层：球场底图 -->
  <g id="court-floor">  木地板底色 </g>
  <g id="court-lines">  边线、中线、三分线、禁区、跳球圈  </g>
  <g id="court-markers"> 篮筐图标 </g>

  <!-- 中层：Hexbin 图层 -->
  <g id="hexbin-left"  transform="translate(0, 0)">   左半场 hexbin  </g>
  <g id="hexbin-right" transform="translate(940, 0)"> 右半场 hexbin (坐标取反) </g>

  <!-- 顶层：UI 叠加（可选） -->
  <g id="ui-overlay"> 中场比分牌、颜色图例 </g>
</svg>
```

数据集的原始坐标以进攻方篮筐为原点。全场图将左篮筐作为左半场的进攻篮筐，右篮筐作为右半场的进攻篮筐。

## 5. 坐标系与半场映射

### 核心思路

数据集中所有投篮记录都以**进攻方篮筐**为原点。全场横向图左右各一个篮筐，需要把数据从"篮筐在下"的坐标系转换到"篮筐在左/右"的坐标系。

**做法：swap X 和 Y 轴，不旋转。**

```
传统视图（篮筐在下方）：          全场横向视图（篮筐在左/右）：

         Y↑                               X 轴（球场宽度）
         │                             ───────────────────
         │  ← 深度方向                  │                 │
    ─────┴─────  X 轴（宽度）           │  左半场  右半场   │
         ● 篮筐                        Y│← ●          ● →│Y
                                       │ 篮筐         篮筐 │
                                       │                 │
                                       ───────────────────
```

### 原始坐标

```
原点 (0, 0) = 进攻方篮筐正下方
X: -250 (左) ~ +250 (右)，球场宽度方向
Y: 0 ~ 470 (从篮筐向外沿纵轴到半场线附近)
```

### D3 渲染坐标交换

**左半场（篮筐在左边，hex 向右展开）：**

```javascript
cx = Y_original;     // 深度 → 水平向右
cy = X_original;     // 宽度 → 垂直方向
```

**右半场（篮筐在右边，hex 向左展开）：**

```javascript
cx = -Y_original;    // 深度 → 水平向左（取负）
cy = X_original;     // 宽度 → 垂直方向（不变）
```

### 最终视觉效果

```
┌──────────────────────────────────────────────────────┐
│                                                      │
│  ◯ 篮筐 ← hex 向右扩展 ──   中场   ── hex 向左扩展 → ◯ 篮筐 │
│                                                      │
│  [左半场数据]                     [右半场数据]         │
│                                                      │
└──────────────────────────────────────────────────────┘
```

- 六边形越靠近中线 = 投篮距离越远（长两分、三分球）
- 六边形越靠近篮筐 = 近距离出手（禁区、篮下）
- 中场完全是空白（无数据覆盖），作为视觉分隔
- 两个半场的 hexbin 对称展开，但数据独立

### JSON 数据无需修改

坐标交换**仅在前端 D3 渲染时进行**。Python 预处理按原始坐标系产出 (x, y)，JSON 结构完全不变。

## 6. JSON 数据集规范

### 文件组织

```
public/data/
  hexbin_season_1997-98.json
  hexbin_season_1998-99.json
  ...
  hexbin_season_2019-20.json   # 共 23 个文件
```

### 单文件结构

```json
{
  "season": "1997-98",
  "league": {
    "hexbins": [
      { "x": 50, "y": 30, "count": 1247, "fg_pct": 0.423 }
    ]
  },
  "teams": {
    "1610612744": {
      "team_name": "Golden State Warriors",
      "abbr": "GSW",
      "hexbins": [
        { "x": 30, "y": 200, "count": 85, "fg_pct": 0.481 }
      ]
    }
  },
  "players": {
    "76001": {
      "player_name": "Stephen Curry",
      "team_id": 1610612744,
      "team_abbr": "GSW",
      "hexbins": [
        { "x": -220, "y": 420, "count": 12, "fg_pct": 0.417 }
      ]
    }
  }
}
```

### 字段规范

| 字段 | 类型 | 约束 |
|---|---|---|
| `season` | string | 固定格式 `"YYYY-YY"` |
| `x` | integer | -250 ~ 250，hex 中心点 X 坐标 |
| `y` | integer | 0 ~ 470，hex 中心点 Y 坐标 |
| `count` | integer | ≥ 1，该六边形内投篮总次数 |
| `fg_pct` | number | 0.0 ~ 1.0，保留 3 位小数，命中次数 / 总次数 |
| `team_id` | integer | NBA 官方 Team ID |
| `team_name` / `abbr` | string | 全名和缩写，用于 UI 展示 |
| `player_name` | string | 全名，用于 UI 展示 |

### 解释

每个 JSON 对象代表一个六边形格子的聚合结果：
- `x`, `y` — hex 中心坐标（D3 hexbin 算法确定）
- `count` — 该六边形内的总出手次数
- `fg_pct` — 该六边形内的命中率 = 命中数 / 总出手数

前端使用方式：
- `count` → 决定六边形大小
- `fg_pct` → 决定颜色深浅（橙红色系）

## 7. 预处理参数

| 参数 | 值 | 说明 |
|---|---|---|
| Hex 半径 | 15 英尺 | 平衡粒度与总格子数 |
| 半场 Y 范围 | 0 ~ 470 | 过滤后场投篮 |
| X 范围 | -250 ~ 250 | 球场宽度边界 |
| 联盟/球队 min count | ≥ 5 | 过滤低频 hex |
| 球员 min count | ≥ 3 | 单球员出手基数较小 |
| 赛季标签格式 | `YYYY-YY` | 如 "1997-98" |
| 单个文件预估大小 | ~435 KB | 联盟 + 30 球队 + 12 球员 |
| 23 文件总计 | ~10 MB | — |

### 赛季定义

```python
# 从 Game Date (YYYYMMDD int) 推导赛季标签
year = game_date // 10000       # 如 19980102 → 1998
month = (game_date // 100) % 100  # → 01
if month >= 10:
    season = f"{year}-{str(year+1)[2:]}"    # "1997-98"
else:
    season = f"{year-1}-{str(year)[2:]}"    # "1997-98"
```

## 8. 前端消费方式

### 数据加载

1. 用户选择赛季 → Pinia Store 检查该赛季 JSON 是否已缓存
2. 未缓存 → `fetch(hexbin_season_{season}.json)` → 写入 Store
3. Store 根据选中层级（league / team / player）提取对应 hexbin 数组
4. 两个半场各有独立的 `seasonScope + scopeType + entityId`，各自提取数据
5. D3 组件监听数据变更，自动重绘

### 颜色标尺同步

两个半场默认使用**动态标尺**——基于左右两侧数据的并集计算 `fg_min` 和 `fg_max`，确保颜色含义一致。提供「固定标尺」开关，切换为 0.0 ~ 1.0 全局基准。

### 时间维度联动

Hexbin 支持与时间-FG%衰减曲线联动，展示特定比赛时段内的投篮空间分布。

**时间字段：** 从 `Period` + `Minutes Remaining` + `Seconds Remaining` 合成比赛绝对秒数（0-2880s，不含加时）。

**时间分箱：** 每节切分前后各 6 分钟，共 8 个 bin：

| Bin | 时段 | 全联盟 FG% | 3PT% | 数据量 |
|---|---|---|---|---|
| 0 | Q1 前6min | 46.9% | 38.5% | 615,335 |
| 1 | Q1 后6min | 46.3% | 37.0% | 608,162 |
| 2 | Q2 前6min | 44.7% | 34.8% | 588,579 |
| 3 | Q2 后6min | 46.3% | 36.3% | 590,400 |
| 4 | Q3 前6min | 45.4% | 36.1% | 587,794 |
| 5 | Q3 后6min | 44.8% | 35.9% | 569,456 |
| 6 | Q4 前6min | 43.5% | 34.1% | 556,445 |
| 7 | Q4 后6min | 43.7% | 33.4% | 572,338 |

**关键发现：** 比赛越临近结束、命中率越低（Q1前6min FG=46.9% → Q4后6min FG=43.7%，落差 3.2pp）。三分命中率衰减最剧烈（-5.1pp），两分球相对抗压（-1.1pp）。平均投篮距离反而增加（11.5→12.6 ft），说明球员疲劳时选择远投但效率降低。

**联动模式：** 衰减曲线不设独立开关。时间 bin 的点击/拖拽作用于当前焦点所在的 Hexbin（跟随用户最后操作过的选择器）。默认焦点在左侧。聚焦的半场按选中时段过滤；非聚焦的半场保持全时段。

**无联动时（默认状态）：** Hexbin 显示全场比赛数据，与原有设计一致。时间曲线作为独立的摘要小图展示 FG% 趋势线。

**JSON 数据扩展：** 每个赛季的 hexbin JSON 中，额外按时间 bin 预聚合。结构扩展为：

```json
{
  "season": "1997-98",
  "league": {
    "hexbins": [
      { "x": 50, "y": 30, "count": 1247, "fg_pct": 0.423 }
    ],
    "hexbins_by_time": {
      "0": [{ "x": 50, "y": 30, "count": 156, "fg_pct": 0.441 }],
      "1": [...],
      "...": "...",
      "7": [...]
    }
  },
  "teams": { "... 同样扩展 hexbins_by_time ..." },
  "players": { "... 同样扩展 hexbins_by_time ..." }
}
```

**前端消费方式：** 时间曲线由 Pinia Store 中的 `selectedTimeBin` 控制（null = 全时段）。Hexbin 组件监听两个状态：`selectedTimeBin === null` → 渲染全时段 `hexbins`；`selectedTimeBin === 3` → 渲染 `hexbins_by_time["3"]`。切换延迟 < 50ms（数据已在内存中）。

**球员级数据量验证：** Curry ~6K-8K 投篮/赛季 → ~750-1000 次/bin。最细的时间 bin 仍可生成有意义的 hexbin（~50-150 个格子）。

## 9. 核验清单

预处理脚本运行完毕后逐项确认：

- [ ] 23 个 JSON 文件全部生成，无空文件
- [ ] 每个文件中 `league.hexbins` 非空
- [ ] 所有 `fg_pct` ∈ [0, 1]
- [ ] 所有 `x` ∈ [-250, 250]，`y` ∈ [0, 470]
- [ ] 所有 `count` ≥ 对应最小阈值
- [ ] 12 位预设球员在有数据的赛季中均存在
- [ ] 总目录大小 ≤ 15 MB
- [ ] 前端 fetch 单个文件 < 200ms（本地环境）
- [ ] 两半场各以各自篮筐为原点渲染，无坐标越界
- [ ] `hexbins_by_time` 在 23 个 JSON 中均包含，无空 bin
- [ ] 时间联动后 hexbin 更新延迟 < 50ms
- [ ] Q4 后 6min (bin==7) 的 hexbin 仍有足够格子数支撑可视化
