第1步：数据理解与清洗
弄清楚每个字段含义，检查缺失值和异常值（重点：X/Y 坐标是否在球场合理范围），验证数据自洽性。
Python + pandas / VS Code
4.7M 条数据，先在小样本上验证逻辑再全量跑。数据质量问题记录下来，并且做一份质检报告。完成后把数据来源、字段说明、处理决策写成一段草稿，避免后面写报告时遗忘。
________________________________________
第2步：顶层设计
2a. Insight 假设与验证
基于数据特征构思 Insight 假设（每条回答：发现什么 + 为什么），然后用 AI 辅助快速验证——将假设和数据摘要提供给 AI，让它从统计角度质疑合理性、补充遗漏视角、发现潜在反例。验证通过后再锁定方向。
2b. 架构设计
确定视图组成（≥2 个联动视图）、页面布局、Store 结构。画出交互关系图（哪个视图触发 → 哪些视图响应）。
2c. 视觉规范
约定一份共享设计规范：色板（主色/辅色/强调色）、图表公共样式（字体、间距、tooltip 格式）、加载态/空数据态/异常态的展示方式。每个视图三种状态（正常/空数据/加载中）在第 3 步开发时就直接实现，不要等联调再补。
2d. Git 协作规范
确定分支策略（每人独立功能分支）、commit 信息格式、合并前 review 流程、.gitignore 内容（node_modules、原始 CSV、.vite 缓存、系统临时文件）。
白板 / Figma / Markdown / AI 辅助验证
先想 Insight 再决定做什么视图。Store 结构、数据格式、视觉规范在这一歩就定死，否则联调大量返工。完成这一步后把架构设计记录下来，直接作为报告素材。
________________________________________
第3步：并行开发
3a. 风险预检
在动手写代码前，把每个视图的设计方案（数据类型、数据量级、交互方式、技术选型）描述给 AI，让它指出可能的性能瓶颈、实现陷阱、ECharts/D3 的坑点。小组讨论后再开始编码。
3b. 数据层
用 Python 清洗原始 CSV，按赛季/区域/球员等维度预聚合，导出为前端可直接 fetch 的 JSON 文件，放入 public/data/ 目录。重点产出一个用于热力图的 hexbin 投篮密度数据和一个球员级汇总表。
不要让前端直接解析 4.7M 条原始数据。JSON 按需拆分（按赛季或按维度分文件），避免单个文件过大导致加载卡顿。
3c. 框架层
基于课程 Vue 模板搭建项目骨架：创建 Pinia Store（按第 2 步约定的结构填充 state、getters、actions），搭建页面布局（顶部控制栏 + 中间视图网格 + 底部详情面板），封装全局过滤器组件（赛季滑块、球队/球员搜索）。在布局中内置第 2 步约定的 CSS 变量和主题配置。
这层由一个人完成，其他两人等待期间可以先在 Notebook 或独立 HTML 里试做图表原型。Store 的 actions 里预留好数据加载和选中状态更新的方法签名。
3d. 视图层	
三人各负责 1-2 个视图组件，独立开发、独立调试。每人用自己视图对应的 JSON 子集做 mock 或直接 fetch，不依赖其他视图即可看到渲染效果。
每人用 ECharts （可能是VueECharts）为主，特殊需求（如 hexbin）可上 D3.js。直接按第 2 步约定的视觉规范实现三种状态（正常/空数据/加载中）。先保证单视图能独立跑通并正确响应 props 变化，不要一开始就接 Store。
工具与配置
Vue 3 + Vite + pnpm + ECharts + vue-echarts + D3.js / Element Plus / Python pandas / Node.js v22.17.0 / Git
先让一个人跑通"数据→框架→一个视图→渲染"的端到端链路验证技术栈，确认没问题再三人铺开。不提交 node_modules。

第4步：整合联动
将所有视图接入全局 Store，实现跨视图联动、全局过滤器、刷选/下钻/动画过渡。
Pinia + Vue DevTools + Chrome Performance 面板
保证单向数据流：操作→Store→所有视图响应。大规模数据切换注意动画流畅度。
________________________________________
第5步：系统检验与 Insight 提炼
用系统探索数据，验证和修正 Insight 假设，对每条发现做归因分析；检查系统功能完整性。
浏览器实战操作 + 截图 + 笔记
Insight = 可视化 + 推理。好的发现往往是交互中意外撞见的。每条 Insight 要有具体数据和原因支撑。验证通过的 Insight 立刻写成一段文字 + 截图，作为报告素材积累，不要等最后再写。
________________________________________
第6步：迭代改进
修 bug、优化交互流畅度、根据 Insight 发现改进或替换视图表达。
同第 3、4 步
优先级：bug > 流畅度 > 表达优化 > 新功能。设硬截止时间，之后只修 bug 不动功能。使用 AI 开发时让它同步记录开发日志（每次改了什么、为什么改、截图对比），并提醒"此处可以用于报告的 XX 部分"，避免事后遗漏关键素材。

第7步：收尾交付
录 Demo、做 PPT（5 分钟）、整理报告（将前几步积累的素材组稿排版为 2 页 + 贡献表）、写 README、打包代码（不含 node_modules）、演练答辩。
现场验证：至少在答辩电脑以外的另一台机器上，执行一次 clone → pnpm install → pnpm dev 全流程，确认环境可迁移、所有功能正常。
